# 秒杀优惠券压测配置指南

## 一、整体架构回顾

秒杀领券有两条路径：

| 路径 | 接口 | 特点 |
|------|------|------|
| 同步 | `POST /api/engine/user-coupon/redeem` | Redis Lua 扣减 → DB 事务（扣库存+插记录+写缓存+发延迟消息） → 返回 |
| 异步 MQ | `POST /api/engine/user-coupon/redeem-mq` | Redis Lua 扣减 → 发 MQ → 返回，DB 操作由消费者异步完成 |

**压测建议：使用 MQ 路径。** 同步路径在高并发下 DB 连接池会成为瓶颈，MQ 路径请求在 Redis 层就返回，吞吐量更高。

---

## 二、配置项详解

### 1. 数据库连接池（HikariCP）— 影响最大

**当前问题：** ShardingSphere 数据源使用 HikariCP，但未配置任何连接池参数，全部走默认值。HikariCP 默认 `maximumPoolSize=10`，即每个数据源最多 10 个数据库连接。

**影响链路：**
- 同步路径：每次领券请求需要 DB 扣库存 + 插入用户优惠券记录，占用 1 个连接
- 10 个连接 = 最多同时处理 10 个 DB 写操作
- 第 11 个请求开始排队等待连接，`connectionTimeout` 默认 30 秒后超时报错
- 压测几百并发时，大量请求卡在等连接上，TPS 上不去，超时报错率飙升

**配置文件：** `engine/src/main/resources/shardingsphere-config.yaml`

**推荐配置：**

```yaml
dataSources:
  ds_0:
    dataSourceClassName: com.zaxxer.hikari.HikariDataSource
    driverClassName: com.mysql.cj.jdbc.Driver
    jdbcUrl: jdbc:mysql://127.0.0.1:3306/mall_coupon_cqupt_0?useUnicode=true&characterEncoding=UTF-8&rewriteBatchedStatements=true&allowMultiQueries=true&serverTimezone=Asia/Shanghai
    username: root
    password: Lqy259931
    # === 连接池配置 ===
    maximumPoolSize: 50        # 最大连接数
    minimumIdle: 10            # 最小空闲连接数
    connectionTimeout: 3000    # 获取连接超时时间(ms)
    idleTimeout: 600000        # 空闲连接存活时间(ms)
    maxLifetime: 1800000       # 连接最大存活时间(ms)
  ds_1:
    # 同上，省略重复
    # ...
```

| 参数 | 默认值 | 推荐值 | 说明 |
|------|--------|--------|------|
| `maximumPoolSize` | 10 | 50 | 池中最大连接数。太小 → 请求排队等连接；太大 → MySQL 连接数耗尽。公式参考：`(CPU核心数 * 2) + 磁盘数`，压测环境可直接设 50 |
| `minimumIdle` | 10 | 10 | 最小空闲连接。保持一定数量的空闲连接避免冷启动延迟 |
| `connectionTimeout` | 30000 | 3000 | 获取连接的等待超时(ms)。默认 30 秒太长，压测时应快速失败，设 3 秒 |
| `idleTimeout` | 600000 | 600000 | 空闲连接多久后被回收(ms)。默认 10 分钟，保持即可 |
| `maxLifetime` | 1800000 | 1800000 | 连接最大存活时间(ms)。默认 30 分钟，避免 MySQL 服务端主动断开 |

### 2. Redis 连接池 — 影响较大

**当前问题：** `application.yaml` 只配了 `host` 和 `port`，未配置连接池。Spring Boot 默认使用 Lettuce 客户端，**默认共享单个 TCP 连接**（通过 Netty 多路复用）。

**影响链路：**
- 秒杀第一步就是执行 Redis Lua 脚本（扣库存 + 记录领取次数）
- 所有并发请求共用一个连接，Lua 脚本虽是原子操作，但命令排队串行执行
- 高并发下 Redis 连接成为瓶颈，Lua 脚本执行延迟增大
- 配合连接池可以让多个请求并行发送到 Redis（Redis 本身支持 pipeline 并行）

**配置文件：** `engine/src/main/resources/application.yaml`

**推荐配置：**

```yaml
spring:
  data:
    redis:
      host: 127.0.0.1
      port: 6379
      lettuce:
        pool:
          max-active: 32        # 最大活跃连接数
          max-idle: 16          # 最大空闲连接数
          min-idle: 8           # 最小空闲连接数
          max-wait: 2000ms      # 获取连接最大等待时间
```

| 参数 | 默认值 | 推荐值 | 说明 |
|------|--------|--------|------|
| `max-active` | -1(无限制) | 32 | 池中最大连接数。太大浪费，太小阻塞。秒杀场景 32 够用 |
| `max-idle` | 8 | 16 | 最大空闲连接。比默认大一些避免频繁创建销毁 |
| `min-idle` | 0 | 8 | 最小空闲连接。保持预热连接，避免冷启动 |
| `max-wait` | -1(无限等待) | 2000ms | 获取连接等待超时。设 2 秒快速失败 |

### 3. RocketMQ 配置 — 影响中等

**当前问题：**
- `send-message-timeout: 2000` — 同步发送超时 2 秒，高并发下 MQ broker 响应慢时会拖慢请求线程
- `retry-times-when-send-failed: 1` — 只重试 1 次，高并发下消息发送失败率上升，重试次数不够容易丢消息
- 缺少发送方式配置，MQ 路径默认同步发送

**影响链路（MQ 路径）：**
- Redis Lua 成功后 → 发 MQ 消息 → 等待 MQ 返回 → 请求结束
- 如果 MQ 发送超时 2 秒，这个请求线程就被阻塞 2 秒
- 线程池耗尽后新请求直接拒绝

**配置文件：** `engine/src/main/resources/application.yaml`

**推荐配置：**

```yaml
rocketmq:
  name-server: common-rocketmq-dev.magestack.cn:9876
  producer:
    group: oneCoupon_engine${unique-name:}-service_common-message-execute_pg
    send-message-timeout: 3000          # 发送超时(ms)
    retry-times-when-send-failed: 2     # 同步发送失败重试次数
    retry-times-when-send-async-failed: 2  # 异步发送失败重试次数
```

| 参数 | 当前值 | 推荐值 | 说明 |
|------|--------|--------|------|
| `send-message-timeout` | 2000 | 3000 | 发送超时(ms)。适当放宽避免网络抖动时大量超时 |
| `retry-times-when-send-failed` | 1 | 2 | 同步发送失败重试次数。多一次重试降低丢消息概率 |
| `retry-times-when-send-async-failed` | 1 | 2 | 异步发送失败重试次数。同上 |

> **进阶优化：** 如果要进一步提升 MQ 路径吞吐量，可以在代码中将 `sendMessage` 改为异步发送（`sendAsync`），请求线程不用等 MQ 返回。

### 4. SQL 日志 — 影响中等

**当前问题：** 两处日志配置都会在压测时产生大量 I/O：

1. `shardingsphere-config.yaml` 中 `sql-show: true` — ShardingSphere 每条 SQL 都打印
2. `application.yaml` 中 `log-impl: org.apache.ibatis.logging.stdout.StdOutImpl` — MyBatis-Plus 每条 SQL 打印到 stdout

**影响链路：**
- 每次 DB 操作产生 2 份日志输出（ShardingSphere + MyBatis）
- 压测时每秒几百次 DB 操作 = 每秒几百行日志写入
- 控制台 I/O 会消耗 CPU 和锁竞争，拖慢整体吞吐量

**推荐配置：**

`shardingsphere-config.yaml`：
```yaml
props:
  sql-show: false   # 压测时关闭
```

`application.yaml`：
```yaml
spring:
  mybatis-plus:
    configuration:
      log-impl: org.apache.ibatis.logging.nologging.NoLoggingImpl   # 压测时关闭 SQL 日志
```

| 配置 | 影响 | 说明 |
|------|------|------|
| `sql-show: true` | 每条 SQL 经过 ShardingSphere 时打印路由、改写、执行信息 | 压测时关闭，平时调试可打开 |
| `log-impl: StdOutImpl` | 每条 SQL 的完整语句和参数打印到控制台 | 改为 `NoLoggingImpl` 关闭 |

### 5. Tomcat 线程池 — 影响较大

**当前问题：** 未配置 Tomcat 线程池参数，走默认值。Spring Boot 默认 `server.tomcat.threads.max=200`。

**影响链路：**
- 每个 HTTP 请求占用 1 个 Tomcat 线程
- 同步路径：线程在等待 DB 连接/DB 执行期间被占用
- MQ 路径：线程在等待 MQ 发送返回期间被占用
- 线程数不够 → 请求排队；线程数太多 → 上下文切换开销大

**配置文件：** `engine/src/main/resources/application.yaml`

**推荐配置：**

```yaml
server:
  port: 10020
  tomcat:
    threads:
      max: 300              # 最大工作线程数
      min-spare: 50         # 最小空闲线程数
    accept-count: 100       # 等待队列长度
    connection-timeout: 5000  # 连接超时(ms)
```

| 参数 | 默认值 | 推荐值 | 说明 |
|------|--------|--------|------|
| `threads.max` | 200 | 300 | 最大工作线程。CPU 密集型不宜太大，IO 密集型可以大些。秒杀场景偏 IO 等待，300 合理 |
| `threads.min-spare` | 10 | 50 | 最小空闲线程。保持预热避免请求来了才创建线程 |
| `accept-count` | 100 | 100 | 等待队列。所有线程忙时新请求进队列，队列满则拒绝。默认 100 可以 |
| `connection-timeout` | 20000 | 5000 | 连接超时(ms)。默认 20 秒太长，压测时 5 秒快速失败 |

---

### 6. JVM 参数 — 影响较大

**当前问题：** 未特别配置 JVM 参数，可能使用默认堆大小。

**影响链路：**
- 高并发下大量对象创建（请求/响应 DTO、JSON 序列化、日志对象等）
- 堆内存不够 → 频繁 GC → STW 停顿 → TPS 锯齿形波动
- 严重时 OOM 直接崩

**推荐 JVM 启动参数：**

```bash
java -Xms2g -Xmx2g \
     -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=100 \
     -XX:G1HeapRegionSize=4m \
     -XX:InitiatingHeapOccupancyPercent=45 \
     -XX:+PrintGCDetails -XX:+PrintGCDateStamps \
     -Xloggc:gc.log \
     -jar engine.jar
```

| 参数 | 说明 |
|------|------|
| `-Xms2g -Xmx2g` | 堆大小固定 2G。避免运行时动态扩缩容带来的停顿 |
| `-XX:+UseG1GC` | 使用 G1 垃圾收集器，适合大堆低延迟场景 |
| `-XX:MaxGCPauseMillis=100` | 目标最大 GC 停顿 100ms |
| `-XX:InitiatingHeapOccupancyPercent=45` | 堆使用 45% 时开始并发标记，避免 Full GC |

---

### 7. MySQL 服务端 — 影响中等

**说明：** 不是应用配置，但直接影响压测结果。

**检查项：**

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| `max_connections` | ≥ 200 | MySQL 最大连接数。两个数据源各 50 + 其他服务，需留余量 |
| `innodb_buffer_pool_size` | 物理内存的 50%-70% | InnoDB 缓冲池，越大越好，减少磁盘 I/O |
| `innodb_flush_log_at_trx_commit` | 2（压测）/ 1（生产） | 设为 2 时每秒刷一次日志，性能提升明显但极端情况可能丢 1 秒数据 |

## 三、配置优先级总结

按影响程度排序，建议按顺序调整：

| 优先级 | 配置项 | 影响程度 | 改动成本 |
|--------|--------|----------|----------|
| 1 | HikariCP 连接池 (`maximumPoolSize`) | 最大 | 改 1 个 yaml 文件 |
| 2 | Redis 连接池 (`lettuce.pool`) | 大 | 改 1 个 yaml 文件 |
| 3 | Tomcat 线程池 | 大 | 改 1 个 yaml 文件 |
| 4 | 关闭 SQL 日志 | 中 | 改 2 个 yaml 文件 |
| 5 | RocketMQ 超时和重试 | 中 | 改 1 个 yaml 文件 |
| 6 | JVM 参数 | 大 | 改启动脚本 |
| 7 | MySQL 服务端参数 | 中 | DBA 操作 |

## 四、压测建议

1. **使用 MQ 路径** (`/api/engine/user-coupon/redeem-mq`)：Redis 层即返回，DB 异步处理，更适合秒杀场景
2. **梯度加压**：从 10 并发开始，逐步增加到 100、500、1000，观察 TPS 和错误率拐点
3. **监控关键指标**：
   - TPS（每秒事务数）
   - RT（响应时间 P50/P90/P99）
   - 错误率（重点关注 401、500、超时）
   - DB 连接池使用率（HikariCP 日志会打印）
   - Redis 连接等待时间
   - GC 频率和停顿时间
4. **预热**：正式压测前先跑 1-2 分钟低并发，让连接池、JIT 编译等预热完成

## 五、本机硬件配置分析

### 你的硬件

| 硬件 | 规格 |
|------|------|
| CPU | Intel i5-14600KF（6 性能核 + 8 能效核，共 14 核 20 线程） |
| GPU | RTX 5060 Ti 16GB |
| 内存 | 32GB |

### 1. CPU — i5-14600KF：压测够用，但有竞争

**架构特点：**
- 6 个 P-core（性能核，支持超线程）= 12 线程
- 8 个 E-core（能效核，不支持超线程）= 8 线程
- 共 14 核 20 线程，单核性能很强（P-core 睿频 5.3GHz）

**压测场景下的竞争：**

本机压测意味着所有组件跑在同一台机器上，它们在抢 CPU：

| 组件 | CPU 占用特点 | 预估占用 |
|------|-------------|----------|
| Engine（Java 应用） | Tomcat 线程处理请求、JWT 解析、JSON 序列化 | 2-4 核 |
| MySQL | SQL 执行、事务、索引计算 | 2-4 核 |
| Redis | Lua 脚本执行、数据读写 | 1-2 核 |
| RocketMQ Broker | 消息存储、投递 | 1-2 核 |
| JMeter（压测工具） | 发送 HTTP 请求、统计结果 | 1-2 核 |
| 操作系统 | 调度、网络栈、文件 I/O | 1 核 |
| **合计** | | **8-15 核** |

**结论：** 14 核 20 线程完全够用。但要注意 P-core 和 E-core 性能差异很大（E-core 性能约为 P-core 的 60-70%），建议将关键进程绑定到 P-core 上：

```powershell
# 将 Engine Java 进程绑定到 P-core（CPU 0-11）
Start-Process java -ArgumentList "-Xms4g","-Xmx4g","-jar","engine.jar" -Priority High

# 或在任务管理器中手动设置亲和性
# Engine → P-core（CPU 0-11）
# MySQL  → P-core（CPU 0-11）
# Redis  → E-core（CPU 12-19）即可，Redis 单线程模型不吃 CPU
```

**实际瓶颈预判：** CPU 不会是瓶颈。压测瓶颈大概率在 DB 连接池或 Redis 连接上。

### 2. 内存 — 32GB：充足，但要合理分配

所有组件跑在一台机器上，内存分配要规划：

| 组件 | 推荐分配 | 说明 |
|------|----------|------|
| 操作系统 + 其他 | 2-3GB | Windows 系统本身占 2-3GB |
| Engine JVM 堆 | 4GB | `-Xms4g -Xmx4g`，秒杀场景对象多，给够 |
| MySQL | 6-8GB | 主要给 `innodb_buffer_pool_size`，尽量把热数据放内存 |
| Redis | 2GB | 默认 `maxmemory` 不限制，建议设上限 |
| RocketMQ Broker | 1-2GB | 消息存储和索引 |
| JMeter | 1-2GB | 压测工具本身也需要内存 |
| **合计** | **16-21GB** | 32GB 绰绰有余 |

**根据你的 32GB 内存调整后的 JVM 参数：**

```bash
java -Xms4g -Xmx4g \
     -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=100 \
     -XX:G1HeapRegionSize=8m \
     -XX:InitiatingHeapOccupancyPercent=40 \
     -jar engine.jar
```

| 参数 | 调整前 | 调整后 | 原因 |
|------|--------|--------|------|
| `-Xms/-Xmx` | 2g | 4g | 32GB 内存充裕，给 JVM 更多空间减少 GC |
| `-XX:G1HeapRegionSize` | 4m | 8m | 堆变大后 region 也应增大，减少 region 管理开销 |
| `-XX:InitiatingHeapOccupancyPercent` | 45 | 40 | 堆变大后提前开始并发标记，避免并发标记来不及 |

**MySQL 配置调整（my.ini / my.cnf）：**

```ini
[mysqld]
innodb_buffer_pool_size = 6G        # 32GB 内存下可给 6GB
innodb_buffer_pool_instances = 6     # buffer_pool 分区数，减少锁竞争
max_connections = 300                # 预留足够连接
innodb_flush_log_at_trx_commit = 2   # 压测模式，每秒刷一次日志
innodb_log_file_size = 1G            # redo log 大一些减少刷盘频率
```

### 3. GPU — RTX 5060 Ti：对压测无影响

**结论：GPU 对 Java 后端压测完全无用。**

- Java 后端应用是纯 CPU + 内存 + I/O 密集型，不涉及 GPU 计算
- JMeter 压测工具也不使用 GPU
- MySQL、Redis、RocketMQ 同理

GPU 只在以下场景有用：
- 机器学习 / 深度学习推理
- 视频编解码
- 图形渲染 / 游戏

**不需要为压测做任何 GPU 配置。**

### 4. 磁盘 — 压测的隐藏瓶颈

你没有提磁盘配置，但磁盘 I/O 对压测影响很大：

| 组件 | 磁盘 I/O 特点 |
|------|--------------|
| MySQL | 写 redo log、binlog、数据文件（即使有 buffer_pool 也要刷盘） |
| Redis | RDB 持久化、AOF 刷盘 |
| RocketMQ | 消息存储到 commitlog、consumequeue |
| 日志文件 | GC log、应用日志 |

**判断方法：**
- 压测时打开任务管理器 → 性能 → 磁盘
- 如果磁盘使用率持续 100%，就是瓶颈

**建议：**
- **NVMe SSD**：以上所有组件放 SSD 上，I/O 不会是瓶颈
- **机械硬盘**：强烈建议将 MySQL 和 RocketMQ 的数据目录移到 SSD
- **压测期间**：可以临时关闭 Redis AOF 持久化（`redis-cli CONFIG SET appendonly no`），减少 I/O

### 6. 压测能达到的量级预估

基于你的硬件配置（本机全组件部署）：

| 并发数 | 预估 TPS | 预估 P99 RT | 瓶颈点 |
|--------|---------|-------------|--------|
| 50 | 2000-4000 | < 50ms | 无明显瓶颈 |
| 200 | 5000-8000 | 50-100ms | DB 连接池开始饱和 |
| 500 | 8000-12000 | 100-300ms | DB 连接池 + Redis 连接池 |
| 1000 | 10000-15000 | 300-800ms | 瓶颈明显，错误率上升 |

> **注意：** 以上为 MQ 路径（异步）的预估值。同步路径因为每个请求都要等 DB 事务完成，TPS 会低 3-5 倍。

如果想压到更高：
