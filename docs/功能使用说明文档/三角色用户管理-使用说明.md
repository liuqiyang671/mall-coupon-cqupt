# 邮惠券商城三角色用户管理使用说明

## 1. 功能范围

本说明面向邮惠券商城的用户认证与个人信息管理功能，覆盖平台人员、商家和用户三个核心角色。

当前实现位于 `merchant-admin` 模块，提供统一账户表、注册、登录、Token 会话校验、个人资料维护、修改密码、退出登录以及商家功能权限边界控制。

## 2. 角色定义

| 角色 | roleType | 业务定位 | 店铺编号 shopNumber |
| --- | ---: | --- | --- |
| 平台人员 | 0 | 平台运营、审核、管理人员 | 不需要 |
| 商家 | 1 | 创建优惠券模板、维护发券任务的商家主体 | 必须具备；注册未传时系统自动生成 |
| 用户 | 2 | 面向邮惠券商城领券、用券的普通用户 | 不需要 |

说明：

- `roleType` 不传时，后端兼容旧商家后台逻辑，默认按商家角色处理。
- 商家专属功能会校验 `roleType = 1` 且当前会话存在 `shopNumber`。

## 3. 数据库说明

### 3.1 建表脚本

统一账户表脚本路径：

```text
resources/database/user_account.sql
```

该脚本创建 `t_user` 表，用于保存平台人员、商家和用户三类账号。

### 3.2 核心字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | bigint | 是 | 用户主键 |
| role_type | tinyint | 是 | 0 平台人员，1 商家，2 用户 |
| shop_number | varchar(64) | 否 | 店铺编号，仅商家角色需要 |
| username | varchar(64) | 是 | 登录用户名 |
| password | varchar(512) | 是 | BCrypt 加密后的密码 |
| nickname | varchar(64) | 否 | 昵称 |
| real_name | varchar(64) | 否 | 真实姓名 |
| phone | varchar(128) | 否 | 手机号 |
| mail | varchar(512) | 否 | 邮箱 |
| avatar_url | varchar(512) | 否 | 头像地址 |
| status | tinyint | 是 | 0 正常，1 禁用 |
| activation_status | tinyint | 是 | 0 未激活，1 已激活 |
| last_login_time | datetime | 否 | 最后登录时间 |
| create_time | datetime | 否 | 创建时间 |
| update_time | datetime | 否 | 修改时间 |
| del_flag | tinyint | 否 | 0 未删除，1 已删除 |

### 3.3 唯一约束

| 索引 | 字段 | 作用 |
| --- | --- | --- |
| uk_role_username | role_type, username | 同一角色下用户名唯一 |
| uk_role_phone | role_type, phone | 同一角色下手机号唯一 |
| uk_role_mail | role_type, mail | 同一角色下邮箱唯一 |
| idx_role_status | role_type, status, activation_status | 提升登录和账号状态筛选效率 |
| idx_shop_number | shop_number | 支持商家数据隔离查询 |

表级约束：

- `role_type` 仅允许 `0`、`1`、`2`。
- `status` 仅允许 `0`、`1`。
- `activation_status` 仅允许 `0`、`1`。

## 4. 接口使用

后端服务默认端口：

```text
http://localhost:10010
```

前端 Vite 代理路径：

```text
/api/merchant-admin -> http://localhost:10010
```

如果前端登录时报 `http proxy error: socket hang up`，优先检查 `merchant-admin` 服务是否已启动、MySQL/Redis 是否可连接、接口路径是否为 `/api/merchant-admin/user/login`。

### 4.1 注册

接口：

```http
POST /api/merchant-admin/user/register
Content-Type: application/json
```

平台人员示例：

```json
{
  "roleType": 0,
  "username": "platform_admin",
  "password": "Admin@123456",
  "nickname": "平台管理员",
  "realName": "张三",
  "phone": "13800138001",
  "mail": "platform@example.com"
}
```

商家示例：

```json
{
  "roleType": 1,
  "username": "merchant001",
  "password": "Admin@123456",
  "nickname": "重邮校园商家",
  "realName": "李四",
  "shopNumber": "1810714735922956666",
  "phone": "13800138002",
  "mail": "merchant@example.com"
}
```

用户示例：

```json
{
  "roleType": 2,
  "username": "customer001",
  "password": "User@123456",
  "nickname": "重邮用户",
  "phone": "13800138003",
  "mail": "customer@example.com"
}
```

响应：

```json
{
  "code": "0",
  "message": "success",
  "data": null
}
```

### 4.2 登录

接口：

```http
POST /api/merchant-admin/user/login
Content-Type: application/json
```

请求示例：

```json
{
  "roleType": 1,
  "username": "merchant001",
  "password": "Admin@123456"
}
```

响应核心字段：

| 字段 | 说明 |
| --- | --- |
| token | JWT Token |
| expireTime | 过期时间，毫秒时间戳 |
| userId | 用户 ID |
| username | 用户名 |
| roleType | 用户角色 |
| nickname | 昵称 |
| shopNumber | 店铺编号，非商家为空 |
| status | 账号状态 |
| activationStatus | 激活状态 |

成功响应示例：

```json
{
  "code": "0",
  "message": "success",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiJ9.xxx",
    "expireTime": 1777118400000,
    "userId": "1",
    "username": "merchant001",
    "roleType": 1,
    "nickname": "重邮校园商家",
    "shopNumber": 1810714735922956666,
    "status": 0,
    "activationStatus": 1
  }
}
```

### 4.3 携带登录态

登录后调用受保护接口时，在请求头携带：

```http
Authorization: Bearer <token>
```

后端会同时校验：

1. JWT 签名和过期时间。
2. Token 是否存在于 Redis 会话缓存。
3. Token 中的 `userId` 与 `roleType` 是否能匹配 Redis 保存的登录态。

### 4.4 查询个人信息

接口：

```http
GET /api/merchant-admin/user/info
Authorization: Bearer <token>
```

响应会返回当前登录用户的角色、昵称、真实姓名、手机号、邮箱、头像、账号状态、激活状态和最后登录时间。

### 4.5 修改个人信息

接口：

```http
POST /api/merchant-admin/user/update
Authorization: Bearer <token>
Content-Type: application/json
```

请求示例：

```json
{
  "phone": "13900139001",
  "mail": "updated@example.com",
  "nickname": "邮惠券商家",
  "realName": "王五",
  "avatarUrl": "https://static.example.com/avatar/merchant001.png"
}
```

说明：

- 手机号和邮箱在同一角色下不能重复。
- 修改时会限定当前登录用户 ID 和 `roleType`，防止跨角色误更新。

### 4.6 修改密码

接口：

```http
POST /api/merchant-admin/user/change-password
Authorization: Bearer <token>
Content-Type: application/json
```

请求示例：

```json
{
  "oldPassword": "Admin@123456",
  "newPassword": "NewPass@789"
}
```

说明：

- 新密码长度要求为 6 到 32 位。
- 新密码不能与原密码相同。
- 修改成功后会删除 Redis 中的当前 Token，用户需要重新登录。

### 4.7 退出登录

接口：

```http
POST /api/merchant-admin/user/logout
Authorization: Bearer <token>
```

退出成功后，Redis 中保存的当前 Token 会被删除。

## 5. 权限控制

### 5.1 公开接口

以下接口不需要 Token：

| 接口 | 说明 |
| --- | --- |
| POST /api/merchant-admin/user/register | 注册 |
| POST /api/merchant-admin/user/login | 登录 |

### 5.2 认证接口

除公开接口外，`merchant-admin` 其他接口均需要 `Authorization: Bearer <token>`。

### 5.3 商家专属接口

以下接口仅商家角色可操作：

| 接口 | 说明 |
| --- | --- |
| POST /api/merchant-admin/coupon-template/save | 创建优惠券模板 |
| GET /api/merchant-admin/coupon-template/page | 分页查询当前商家模板 |
| GET /api/merchant-admin/coupon-template/find | 查询当前商家模板详情 |
| POST /api/merchant-admin/coupon-template/terminate | 结束当前商家模板 |
| POST /api/merchant-admin/coupon-template/increase-number | 增加当前商家模板发行量 |
| POST /api/merchant-admin/coupon-task/create | 创建优惠券推送任务 |
| GET /api/merchant-admin/coupon-task/page | 分页查询当前商家推送任务 |
| GET /api/merchant-admin/coupon-task/find | 查询当前商家推送任务详情 |

权限规则：

1. 当前用户必须为 `roleType = 1`。
2. 当前登录态必须包含 `shopNumber`。
3. 查询和修改商家数据时，后端会按 `shopNumber` 过滤，避免横向越权。

## 6. Redis 会话与风控

| Key 模板 | 说明 |
| --- | --- |
| one-coupon_merchant-admin:user:token:%s:%s | 用户 Token，参数为 `roleType` 和 `userId` |
| one-coupon_merchant-admin:user:login-fail:%s:%s | 登录失败次数，参数为 `roleType` 和 `username` |
| one-coupon_merchant-admin:user:login-lock:%s:%s | 登录锁定标识，参数为 `roleType` 和 `username` |

登录风控规则：

- 同一角色、同一用户名连续登录失败达到 5 次后锁定 30 分钟。
- 登录成功后清理失败次数。
- 角色维度参与 Redis Key，允许同名账号分别存在于平台人员、商家、用户三个角色中。

## 7. 常见错误

| 场景 | 错误提示 | 处理建议 |
| --- | --- | --- |
| 用户名或密码为空 | 用户名和密码不能为空 | 检查登录表单必填项 |
| 密码错误或账号不存在 | 用户名或密码错误 | 不暴露账号是否存在，前端统一提示 |
| 连续登录失败 | 账号已被锁定，请30分钟后再试 | 等待锁定过期或由管理员处理 |
| 账号禁用 | 账号已被禁用，请联系管理员 | 平台人员恢复账号状态 |
| 账号未激活 | 账号未激活，请先完成账户激活 | 完成激活流程或后台激活 |
| Token 失效 | 未登录或登录已过期 | 清理本地登录态并跳转登录页 |
| 非商家调用商家接口 | 当前功能仅商家角色可操作 | 切换商家账号或隐藏入口 |

## 8. 前端接入建议

1. 登录页增加角色选择控件，提交 `roleType`、`username`、`password`。
2. 注册页根据角色动态展示字段：商家展示 `shopNumber`，平台人员和用户不展示。
3. 登录成功后保存 `token`、`userId`、`username`、`roleType`、`shopNumber`、`expireTime`。
4. 请求拦截器统一注入 `Authorization: Bearer <token>`。
5. 路由守卫根据 `roleType` 控制菜单：
   - 平台人员：进入平台管理页面。
   - 商家：进入优惠券模板和推送任务管理页面。
   - 用户：进入领券、券包、个人中心页面。
6. 响应 401 时清理本地登录态，并跳转登录页。

## 10. 本地启动检查清单

1. 执行 `resources/database/user_account.sql` 创建或更新统一账户表。
2. 确认 MySQL 已创建 `mall_coupon_cqupt_0` 和 `mall_coupon_cqupt_1`。
3. 确认 Redis `127.0.0.1:6379` 可连接，密码与 `merchant-admin/src/main/resources/application.yaml` 一致。
4. 启动 `merchant-admin`，默认监听 `10010`。
5. 启动前端后，通过 `/api/merchant-admin/user/login` 登录。
6. 如果出现 Vite 代理 `socket hang up`，先直接访问后端健康状态或接口文档，确认 `merchant-admin` 没有启动失败。
