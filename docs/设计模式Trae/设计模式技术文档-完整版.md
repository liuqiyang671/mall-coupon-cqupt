## 2. 设计模式理论基础

### 2.1 设计模式的概念

**设计模式（Design Pattern）** 是指在软件设计中反复出现的、经过验证的、可复用的解决方案模板。

#### 2.1.1 设计模式的四大要素

1. **模式名称**：助记符，用于描述设计问题、解决方案和效果
2. **问题**：描述何时适用该模式，说明问题产生的背景
3. **解决方案**：描述设计的组成成分、职责和协作关系
4. **效果**：模式应用的优点和缺点，影响分析

#### 2.1.2 为什么使用设计模式？

- 提高代码复用性
- 增强可维护性
- 提升可扩展性
- 促进团队沟通
- 降低耦合度

### 2.2 设计模式的分类

根据目的可分为三大类：

#### 2.2.1 创建型模式（5 种）

| 模式 | 意图 | 适用场景 |
|------|------|---------|
| 工厂方法 | 定义创建对象的接口，让子类决定实例化 | 需要灵活扩展新产品类型 |
| 抽象工厂 | 创建相关或依赖对象的家族 | 需要创建一系列相关产品 |
| 单例 | 保证一个类仅有一个实例 | 全局配置、资源管理器 |
| 建造者 | 分步骤构建复杂对象 | 对象属性众多 |
| 原型 | 通过复制现有对象创建新对象 | 创建对象成本高 |

#### 2.2.2 结构型模式（7 种）

| 模式 | 意图 | 适用场景 |
|------|------|---------|
| 适配器 | 转换接口使兼容 | 复用现有类但接口不匹配 |
| 装饰器 | 动态添加职责 | 需要动态添加功能 |
| 代理 | 控制对象访问 | 延迟加载、访问控制 |
| 外观 | 提供统一接口 | 简化复杂子系统 |

#### 2.2.3 行为型模式（11 种）

| 模式 | 意图 | 适用场景 |
|------|------|---------|
| 策略 | 封装可互换的算法 | 多种算法可替换 |
| 模板方法 | 定义算法骨架 | 算法流程固定，部分可变 |
| 观察者 | 定义一对多依赖 | 事件驱动、消息通知 |
| 责任链 | 多个对象处理请求 | 多级审批、过滤器链 |

### 2.3 设计模式的核心原则（SOLID）

#### 2.3.1 单一职责原则（SRP）

> 一个类应该只有一个引起它变化的原因。

```java
// ✅ 正确：每个策略类只负责一种优惠券计算
public class DiscountCalculationStrategy implements CouponCalculationStrategy {
    @Override
    public BigDecimal calculateDiscount(CouponTemplateDO template, BigDecimal orderAmount) {
        // 只负责折扣券的计算逻辑
        DiscountCouponDO discountCoupon = (DiscountCouponDO) template;
        return orderAmount.multiply(BigDecimal.valueOf(discountCoupon.getDiscountRate()));
    }
}
```

#### 2.3.2 开闭原则（OCP）

> 软件实体应该对扩展开放，对修改关闭。

- 新增功能时，通过扩展已有代码实现
- 避免修改已有代码

#### 2.3.3 里氏替换原则（LSP）

> 子类对象必须能够替换掉所有父类对象。

```java
// ✅ 所有策略可互换
CouponCalculationStrategy strategy1 = new DiscountCalculationStrategy();
CouponCalculationStrategy strategy2 = new ThresholdCalculationStrategy();
```

#### 2.3.4 接口隔离原则（ISP）

> 客户端不应该依赖它不需要的接口。

#### 2.3.5 依赖倒置原则（DIP）

> 高层模块不应该依赖低层模块，二者都应该依赖其抽象。

```java
// ✅ 依赖抽象接口
@Service
@RequiredArgsConstructor
public class CouponCalculationService {
    private final CouponCalculationStrategy strategy;  // 依赖抽象
}
```

## 3. 创建型设计模式应用

### 3.1 工厂模式（Factory Pattern）

#### 3.1.1 模式讲解

**工厂模式**提供了一种创建对象的最佳方式，将对象的创建和使用分离。

| 类型 | 特点 | 适用场景 |
|------|------|---------|
| 简单工厂 | 一个工厂类创建所有对象 | 产品类型少 |
| 工厂方法 | 每个产品对应一个工厂类 | 产品类型多 |
| 抽象工厂 | 创建产品族的工厂接口 | 需要创建相关产品族 |

#### 3.1.2 项目应用实例

**应用场景**：优惠券类型对象创建

**代码位置**：
- `settlement/src/main/java/com/cqupt/settlement/common/util/CouponFactory.java`

**核心代码**：

```java
/**
 * 优惠券工厂类 - 简单工厂模式实现
 * 根据优惠券类型创建对应的优惠券对象
 */
public class CouponFactory {
    
    /**
     * 创建优惠券对象
     */
    public static CouponTemplateDO createCoupon(
            CouponTemplateDO coupon, 
            Map<String, Object> additionalParams) {
        
        // 参数验证
        if (coupon.getType() == null || 
            coupon.getType() >= DiscountTypeEnum.values().length || 
            coupon.getType() < 0) {
            throw new IllegalArgumentException("Invalid coupon type");
        }
        
        // 根据枚举类型创建不同优惠券对象
        switch (DiscountTypeEnum.values()[coupon.getType()]) {
            case FIXED_DISCOUNT:
                Integer fixedDiscountAmount = (Integer) additionalParams.get("discountAmount");
                return new FixedDiscountCouponDO(coupon, fixedDiscountAmount);
                
            case THRESHOLD_DISCOUNT:
                Integer thresholdAmount = (Integer) additionalParams.get("thresholdAmount");
                Integer thresholdDiscountAmount = (Integer) additionalParams.get("discountAmount");
                return new ThresholdCouponDO(coupon, thresholdAmount, thresholdDiscountAmount);
                
            case DISCOUNT_COUPON:
                Double discountRate = (Double) additionalParams.get("discountRate");
                return new DiscountCouponDO(coupon, discountRate);
                
            default:
                throw new IllegalArgumentException("Invalid coupon type");
        }
    }
}
```

**优惠券类型枚举**：

```java
public enum DiscountTypeEnum {
    FIXED_DISCOUNT(0, "立减券"),
    THRESHOLD_DISCOUNT(1, "满减券"),
    DISCOUNT_COUPON(2, "折扣券");
    
    private final Integer code;
    private final String description;
    
    DiscountTypeEnum(Integer code, String description) {
        this.code = code;
        this.description = description;
    }
}
```

#### 3.1.3 应用效果分析

**解决的问题**：

1. 对象创建解耦 - 调用方无需知道具体创建细节
2. 集中管理创建逻辑 - 所有创建逻辑集中在工厂类
3. 符合开闭原则 - 新增类型只需扩展 switch 分支

**架构优势**：

| 优势 | 说明 |
|------|------|
| 简化调用 | 调用方无需关心对象创建细节 |
| 易于扩展 | 新增优惠券类型只需修改工厂类 |
| 代码复用 | 创建逻辑集中，避免重复代码 |
| 类型安全 | 通过枚举类型检查 |

**使用示例**：

```java
Map<String, Object> params = new HashMap<>();
params.put("discountRate", 0.8);

CouponTemplateDO coupon = CouponFactory.createCoupon(baseTemplate, params);
BigDecimal finalAmount = coupon.calculateDiscount(orderAmount);
```

#### 3.1.4 适用场景总结

**何时使用**：

- 系统需要根据不同类型创建不同对象
- 对象创建逻辑复杂，需要集中管理
- 希望调用方无需关心对象创建细节

**最佳实践**：

1. 配合策略模式使用
2. 使用枚举替代魔法值
3. 考虑使用工厂方法模式（产品类型很多时）

### 3.2 单例模式（Singleton Pattern）

#### 3.2.1 模式讲解

**单例模式**确保一个类只有一个实例，并提供全局访问点。

| 实现方式 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| 饿汉式 | 线程安全，实现简单 | 可能浪费内存 | 实例一定会使用 |
| 懒汉式（双重检查） | 延迟加载，线程安全 | 实现复杂 | 实例可能不使用 |
| 静态内部类 | 延迟加载，线程安全，简洁 | - | 推荐方式 |
| 枚举 | 线程安全，防止反射攻击 | 不够灵活 | 需要序列化 |

#### 3.2.2 项目应用实例

**应用场景 1**：Hutool Singleton 管理 Lua 脚本对象

**代码位置**：
- `distribution/src/main/java/.../ReadExcelDistributionListener.java`

```java
// 使用 Hutool 的 Singleton 管理 Lua 脚本对象
// 确保在整个程序的运行期间，某个对象在内存中永远只被创建一次
DefaultRedisScript<Long> buildLuaScript = Singleton.get(
    STOCK_DECREMENT_AND_BATCH_SAVE_USER_RECORD_LUA_PATH, 
    () -> {
        // 1. 创建 Spring Data Redis 提供的 Lua 脚本操作对象
        DefaultRedisScript<Long> redisScript = new DefaultRedisScript<>();
        
        // 2. 指定 Lua 脚本的物理文件来源
        redisScript.setScriptSource(
            new ResourceScriptSource(
                new ClassPathResource(STOCK_DECREMENT_AND_BATCH_SAVE_USER_RECORD_LUA_PATH)
            )
        );
        
        // 3. 强制设置脚本的返回值类型
        redisScript.setResultType(Long.class);
        
        // 4. 返回完整装配好的脚本对象
        // 这个对象会被 Hutool 的 Singleton 内部缓存起来
        // 下次执行这行代码时，会直接从缓存拿
        return redisScript;
    }
);
```

**应用场景 2**：Spring 容器管理的单例 Bean

```java
@Configuration
@RequiredArgsConstructor
public class CacheConfiguration {
    
    private final RedisDistributedProperties redisDistributedProperties;
    
    /**
     * 创建 Redis Key 序列化器
     * Spring Bean 默认为 singleton 作用域
     */
    @Bean
    public RedisKeySerializer redisKeySerializer() {
        String prefix = Optional.ofNullable(redisDistributedProperties.getPrefix()).orElse("");
        String prefixCharset = redisDistributedProperties.getPrefixCharset();
        return new RedisKeySerializer(prefix, prefixCharset);
    }
}
```

#### 3.2.3 应用效果分析

**解决的问题**：

1. 避免重复加载 - Lua 脚本只需加载一次
2. 节省内存资源 - 整个应用生命周期内只存在一个实例
3. 线程安全 - Hutool Singleton 内部使用线程安全的 ConcurrentHashMap

**架构优势**：

| 优势 | 说明 |
|------|------|
| 内存优化 | 避免重复创建对象 |
| 性能提升 | 直接从缓存获取 |
| 线程安全 | 内部实现线程安全 |
| 资源管理 | 统一管理系统级共享资源 |

#### 3.2.4 适用场景总结

**何时使用**：

- 需要频繁创建且消耗资源较大的对象
- 全局配置对象、工具类对象
- 需要跨模块共享的无状态服务对象

**最佳实践**：

1. 优先使用 Spring 容器管理单例
2. 使用 Hutool Singleton 工具类
3. 注意线程安全问题

### 3.3 建造者模式（Builder Pattern）

#### 3.3.1 模式讲解

**建造者模式**将复杂对象的构建与其表示分离，使得同样的构建过程可以创建不同的表示。

**核心思想**：
- 分步骤构建复杂对象
- 隔离构建过程与最终表示
- 支持链式调用，提高可读性

#### 3.3.2 项目应用实例

3. 参数名称明确 - 不易混淆

**架构优势**：

| 优势 | 说明 |
|------|------|
| 可读性强 | 链式调用，参数名明确 |

- 需要构建不可变对象
- 对象创建过程需要分步骤、可配置

1. 配合 Lombok 使用，减少样板代码
2. 处理继承关系时注意特殊处理
3. 在 build() 方法中进行参数校验

## 4. 结构型设计模式应用

### 4.1 代理模式（Proxy Pattern）

#### 4.1.1 模式讲解

**代理模式**为其他对象提供一种代理以控制对这个对象的访问。

| 类型 | 特点 | 应用场景 |
|------|------|---------|
| 静态代理 | 代理类和目标类实现同一接口 | 接口固定 |
| 动态代理（JDK） | 运行时动态生成代理类 | Spring AOP 默认 |
| 动态代理（CGLIB） | 通过继承实现代理 | 可以代理类 |
| 虚拟代理 | 延迟加载 | 加载大图片 |
| 缓存代理 | 缓存结果 | 数据库查询缓存 |

#### 4.1.2 项目应用实例

**应用场景 1**：AOP 动态代理实现幂等性控制

**代码位置**：
- `framework/src/main/java/com/mall/cqupt/framework/idempotent/NoDuplicateSubmitAspect.java`

```java
/**
 * 防止用户重复提交表单信息切面控制器
 * 使用 Spring AOP 动态代理实现
 */
@Aspect
@RequiredArgsConstructor
public final class NoDuplicateSubmitAspect {
    
    private final RedissonClient redissonClient;
    
    /**
     * 增强方法标记 @NoDuplicateSubmit 注解逻辑
     */
    @Around("@annotation(com.mall.cqupt.framework.idempotent.NoDuplicateSubmit)")
    public Object noRepeatSubmit(ProceedingJoinPoint joinPoint) throws Throwable {
        // 获取注解
        NoDuplicateSubmit noDuplicateSubmit = getNoRepeatSubmitAnnotation(joinPoint);
        
        // 构建分布式锁 Key
        String lockKey = String.format(
            "no-repeat-submit:path:%s:currentUserId:%s:md5:%s", 
            getServletPath(), getCurrentUserId(), calcArgsMD5(joinPoint)
        );
        
        // 获取分布式锁
        RLock lock = redissonClient.getLock(lockKey);
        
        // 尝试获取锁，获取锁失败就意味着已经重复提交，直接抛出异常
        if (!lock.tryLock()) {
            throw new ClientException(noDuplicateSubmit.message());
        }
        
        Object result;
        try {
            // 执行标记了防重复提交注解的方法原逻辑
            result = joinPoint.proceed();
        } finally {
            lock.unlock();
        }
        return result;
    }
    
    /**
     * 计算参数 MD5 值
     */
    private String calcArgsMD5(ProceedingJoinPoint joinPoint) {
        return DigestUtil.md5Hex(JSON.toJSONBytes(joinPoint.getArgs()));
    }
}
```

**使用示例**：

```java

@RequiredArgsConstructor

    /**
     * 领取优惠券
     * 使用 @NoDuplicateSubmit 防止重复提交
     */

    public Result<Void> receiveCoupon(@RequestBody ReceiveCouponReqDTO request) {
        couponService.receiveCoupon(request);
        return Results.success();
    }
}
```

```java
        EngineRedisConstant.COUPON_TEMPLATE_KEY, 
        requestParam.getCouponTemplateId()
    );
    Map<Object, Object> couponTemplateCacheMap = stringRedisTemplate.opsForHash().entries(couponTemplateCacheKey);
    
    // 2. 如果存在直接返回，不存在需要通过布隆过滤器、缓存空值以及双重判定锁的形式读取数据库
    if (MapUtil.isEmpty(couponTemplateCacheMap)) {
        // 3. 判断布隆过滤器是否存在指定模板 ID
        if (!couponTemplateQueryBloomFilter.contains(requestParam.getCouponTemplateId())) {
            throw new ClientException("优惠券模板不存在");
        }
        
        // 4. 获取分布式锁，防止缓存击穿
        RLock lock = redissonClient.getLock(

        );
        lock.lock();
        
        try {
            // 5. 双重判定锁 + 查询数据库（真实主题）
            CouponTemplateDO couponTemplateDO = couponTemplateMapper.selectOne(queryWrapper);
            
            // 6. 写入 Redis 缓存
            // ... 省略具体实现
        } finally {
            lock.unlock();
        }
    }
    
    return BeanUtil.mapToBean(couponTemplateCacheMap, CouponTemplateQueryRespDTO.class, false, CopyOptions.create());
}
```

### 5.1 策略模式（Strategy Pattern）

#### 5.1.1 模式讲解

**策略模式**定义了一系列算法，并将每个算法封装起来，使它们可以相互替换。

**核心思想**：
- 将算法的定义和使用分离

```
├─────────────────┤

         │ uses
         ▼
┌─────────────────┐
│   Strategy     │  ← 抽象策略接口
├─────────────────┤
│ + algorithm()  │
└────────┬────────┘
         │
    ┌────┼────┐
    │    │    │
┌───▼──┐┌─▼───┐┌▼─────┐
│StrategyA│ │StrategyB│ │StrategyC│
└──────┘└─────┘└──────┘
```

```java
/**
 * 折扣券计算策略
 * 例如：折扣率 0.8，订单金额 100 元，优惠后金额为 80 元
 */
public class DiscountCalculationStrategy implements CouponCalculationStrategy {
    @Override
    public BigDecimal calculateDiscount(CouponTemplateDO template, BigDecimal orderAmount) {
        DiscountCouponDO discountCoupon = (DiscountCouponDO) template;
        return orderAmount.multiply(BigDecimal.valueOf(discountCoupon.getDiscountRate()));
    }
}

/**
 * 满减券计算策略
 * 例如：满 100 减 50，订单金额 150 元，优惠金额为 50 元
 */
public class ThresholdCalculationStrategy implements CouponCalculationStrategy {
    @Override
    public BigDecimal calculateDiscount(CouponTemplateDO template, BigDecimal orderAmount) {
        ThresholdCouponDO thresholdCoupon = (ThresholdCouponDO) template;
        if (orderAmount.compareTo(BigDecimal.valueOf(thresholdCoupon.getThresholdAmount())) >= 0) {
            return BigDecimal.valueOf(thresholdCoupon.getDiscountAmount());
        }
        return BigDecimal.ZERO;
    }
}

/**
 * 立减券计算策略
 * 例如：立减 50 元，订单金额无论多少，优惠金额都是 50 元
 */
public class FixedDiscountCalculationStrategy implements CouponCalculationStrategy {
    @Override
    public BigDecimal calculateDiscount(CouponTemplateDO template, BigDecimal orderAmount) {
        FixedDiscountCouponDO fixedDiscount = (FixedDiscountCouponDO) template;
        return BigDecimal.valueOf(fixedDiscount.getDiscountAmount());
    }
}
```

**上下文类**：

```java
/**
 * 优惠券计算服务 - 上下文角色
 */
@Service
public class CouponCalculationService {
    
    /**
     * 计算优惠金额
     * @param coupon 具体的优惠券实例
     * @param orderAmount 订单金额
     * @return 计算出的优惠金额
     */
    public BigDecimal calculateDiscount(CouponTemplateDO coupon, BigDecimal orderAmount) {
        // 根据优惠券类型动态选择策略
        CouponCalculationStrategy strategy = getCouponCalculationStrategy(coupon);
        return strategy.calculateDiscount(coupon, orderAmount);
    }
    
    private CouponCalculationStrategy getCouponCalculationStrategy(CouponTemplateDO coupon) {
        switch (DiscountTypeEnum.values()[coupon.getType()]) {
            case DISCOUNT_COUPON:
                return new DiscountCalculationStrategy();
            case THRESHOLD_DISCOUNT:
                return new ThresholdCalculationStrategy();
            case FIXED_DISCOUNT:
                return new FixedDiscountCalculationStrategy();
            default:
                throw new IllegalArgumentException("Invalid coupon type");
        }
    }
}
```

### 5.2 模板方法模式（Template Method Pattern）

```java
/**
 * RocketMQ 抽象公共发送消息组件
 * 模板方法模式实现
 */
@RequiredArgsConstructor
@Slf4j(topic = "CommonSendProduceTemplate")
public abstract class AbstractCommonSendProduceTemplate<T> {
    
    private final RocketMQTemplate rocketMQTemplate;
    
    /**
     * 构建消息发送事件基础扩充属性实体
     * 子类必须实现
     */
    protected abstract BaseSendExtendDTO buildBaseSendExtendParam(T messageSendEvent);
    
    /**
     * 构建消息基本参数，请求头、Keys...
     * 子类必须实现
     */
    protected abstract Message<?> buildMessage(T messageSendEvent, BaseSendExtendDTO requestParam);
    
    /**
     * 消息事件通用发送 - 模板方法
     */
    public SendResult sendMessage(T messageSendEvent) {
        return sendMessage(messageSendEvent, null);
    }
    
    /**
     * 消息事件通用发送 - 模板方法核心逻辑
     */
    @SneakyThrows
    public SendResult sendMessage(T messageSendEvent, Long deliverTimeStamp) {
        // 1. 调用子类方法构建基础参数
        BaseSendExtendDTO baseSendExtendDTO = buildBaseSendExtendParam(messageSendEvent);
        
        // 2. 调用子类方法构建消息
        Message<?> message = buildMessage(messageSendEvent, baseSendExtendDTO);
        
        // 3. 统一的消息发送逻辑（固定不变）
        SendResult sendResult = rocketMQTemplate.syncSend(
            destinationBuilder.toString(),
            message,
            baseSendExtendDTO.getSentTimeout(),
            Optional.ofNullable(baseSendExtendDTO.getDelayLevel()).orElse(0)
        );
        
        log.info("[生产者] {} - 发送结果：{}", baseSendExtendDTO.getEventName(), ...);
        return sendResult;
    }
}
```

#### 5.2.3 应用效果分析

**解决的问题**：

1. 代码复用 - 公共逻辑提取至父类
2. 扩展点控制 - 子类只能扩展特定步骤
3. 统一日志异常 - 集中管理

**架构优势**：

| 优势 | 说明 |
|------|------|
| 代码复用 | 公共逻辑在父类 |
| 易于扩展 | 只需重写特定方法 |
| 统一管控 | 日志、异常统一处理 |

#### 5.4.1 模式讲解

#### 5.4.2 项目应用实例

```java
/**
 * 全局异常处理器
 * 责任链模式：异常按优先级依次传递，直到被处理
 */
@Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    /**
     * 责任链节点 1：拦截参数验证异常
     */
    @SneakyThrows
    @ExceptionHandler(value = MethodArgumentNotValidException.class)
    public Result validExceptionHandler(HttpServletRequest request, MethodArgumentNotValidException ex) {
        BindingResult bindingResult = ex.getBindingResult();
        FieldError firstFieldError = CollectionUtil.getFirst(bindingResult.getFieldErrors());
        String exceptionStr = Optional.ofNullable(firstFieldError)
                .map(FieldError::getDefaultMessage)
                .orElse(StrUtil.EMPTY);
        log.error("[{}] {} [ex] {}", request.getMethod(), getUrl(request), exceptionStr);
        return Results.failure(BaseErrorCode.CLIENT_ERROR.code(), exceptionStr);
    }
    
    /**
     * 责任链节点 2：拦截应用内抛出的异常
     */
    @ExceptionHandler(value = {AbstractException.class})
    public Result abstractException(HttpServletRequest request, AbstractException ex) {
        log.error("[{}] {} [ex] {}", request.getMethod(), request.getRequestURL().toString(), ex, ex.getCause());
        return Results.failure(ex);
    }
    
    /**
     * 责任链节点 3：拦截未捕获异常（兜底）
     */
    @ExceptionHandler(value = Throwable.class)
    public Result defaultErrorHandler(HttpServletRequest request, Throwable throwable) {
        log.error("[{}] {} ", request.getMethod(), getUrl(request), throwable);
        return Results.failure();
    }
}
```

#### 5.4.3 应用效果分析

**解决的问题**：

1. 多级处理 - 异常按优先级依次传递
2. 解耦请求与处理 - 无需知道最终谁处理
3. 动态组合 - 可动态调整处理顺序

#### 5.4.4 适用场景总结

**何时使用**：

- 多个对象可处理同一请求
- 需要按顺序执行多个处理步骤
- 希望动态指定处理者集合

| 适配器模式 | ⭐⭐⭐ | distribution | 接口兼容，复用现有类 |
| 责任链模式 | ⭐⭐⭐ | framework | 多级处理，动态组合 |

### 7.2 最佳实践建议

#### 7.2.1 何时选择何种模式？

| 业务需求 | 推荐模式 | 实施方法 |
|---------|---------|---------|
| 多种算法可替换 | 策略模式 | 定义策略接口 + 具体实现 + 上下文 |
| 对象创建复杂 | 工厂模式 + 建造者模式 | 工厂负责选择，建造者负责构建 |
| 需要横切逻辑 | 代理模式（AOP） | 使用 Spring AOP + 自定义注解 |
| 事件驱动架构 | 观察者模式 | 消息队列 + 事件类 + 监听器 |
| 算法骨架固定 | 模板方法模式 | 抽象类定义流程，子类实现细节 |
| 多级处理流程 | 责任链模式 | Spring 拦截器链或过滤器链 |
| 全局唯一对象 | 单例模式 | Spring Bean 或 Hutool Singleton |

#### 7.2.2 避免过度设计

⚠️ **注意事项**：

1. 不要为了用模式而用模式
2. 简单场景直接使用 if-else 更直观
3. 模式是手段，不是目的
4. 优先使用语言特性（Stream、Optional）

#### 7.2.3 重构建议

如果代码出现以下"代码异味"，考虑引入设计模式：

| 代码异味 | 推荐模式 |
|---------|---------|
| 大量 if-else 或 switch | 策略模式 |
| 重复的创建逻辑 | 工厂模式 |
| 横切逻辑重复 | 代理模式（AOP） |
| 对象构建复杂 | 建造者模式 |
| 类之间耦合严重 | 观察者模式 |

### 7.3 常见误区与避免方法

1. **过早抽象**
   - ❌ 场景还没确定就设计复杂模式
   - ✅ 先简单实现，等需求稳定后再重构

2. **过度使用**
   - ❌ 所有地方都用设计模式
   - ✅ 简单代码直接写，模式用于复杂场景

3. **死板套用**
   - ❌ 严格按照模式结构，不考虑实际情况
   - ✅ 理解模式本质，灵活变通

4. **忽视测试**
   - ❌ 引入模式后不写测试
   - ✅ 每个模式都应该可测试
